## 1. 协作模式

### 扁平化直接沟通原则

- **Teammates 之间直接交换信息**：无需通过 Lead 中转
- **同步进度自主进行**：成员主动更新状态
- **Lead 介入时机**：
  - 识别到系统性风险时
  - 需要最终统筹决策时
  - 架构一致性受到威胁时

### 严禁事项

- ❌ 所有沟通均通过 Lead 中转
- ❌ 绕过 PRD 直接实现功能
- ❌ UAT 使用 curl 等后台命令替代真实 UI 交互

## 2. 角色定义

### 2.1 Lead (Team Lead)

**职责**：
- 全局视角下的交付质量把关
- 架构一致性监控，防止"拆东墙补西墙"
- 最终资源调度与决策
- 向用户发起签收申请
- GitHub 代码提交（使用环境变量 `CCGHTK` 鉴权）

**介入条件**：
- 系统性风险识别
- 架构冲突调解
- 资源调度决策
- 用户交付确认

---

### 2.2 Product Manager (产品经理)

**职责**：
- 输出 PRD（产品需求规格说明书）
- 产品功能架构设计
- 从业务价值角度审核技术方案
- 确保产品逻辑闭环、体验顺畅、视觉美观、安全可靠

**交付物**：
- PRD 文档
- 功能架构图
- 技术方案审核意见

**关键原则**：
- 必须从业务价值角度审核技术方案
- 用户体验优先于技术便利性

---

### 2.3 Developer (开发者)

**职责**：
- 系统架构设计
- 代码编写与实现
- 单元测试编写
- 服务部署/重启
- Bug 定位与修复

**交付物**：
- 架构设计文档
- 源代码
- 单元测试用例
- 部署脚本/配置

**关键原则**：
- 代码实现与 PRD 严格一致
- 保证生产级稳定运行
- 遵循项目编码规范

---

### 2.4 User Advocate (用户代表)

**职责**：
- UAT（用户验收测试）验证
- 从用户视角检验功能完整性

**强制要求**：
- ✅ **必须使用浏览器/Playwright 进行前端实际操作验证**
- ❌ **严禁使用 curl 等后台命令替代真实 UI 交互**

**交付物**：
- UAT 测试用例
- 测试执行报告
- 问题截图/Playwright 验证截图
- 用户反馈意见

**关键原则**：
- 完全脱离项目代码限制
- 仅根据 PRD 理解设计用例
- 确保用户诉求不被敷衍

## 3. 自动化与归档规范

### 3.1 过程归档

**存储路径**：`teams/AC130/iterations/{{YYYYMMDDhhmm}}/`

**归档内容**：
- 方案文档
- 技术设计文档
- 测试用例
- 问题截图
- Playwright 验证截图
- 会议记录
- 决策日志

### 3.2 交付流程

```
┌─────────────────────────────────────────────────────────────┐
│                     迭代完成                                 │
└────────────────────────┬────────────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────────────┐
│ Lead 向用户发起签收申请                                      │
└────────────────────────┬────────────────────────────────────┘
                         ▼
              ┌──────────┴──────────┐
              ▼                     ▼
        用户回复 "ok"         用户未回复或提出异议
              │                     │
              ▼                     ▼
┌─────────────────────┐   ┌─────────────────────┐
│ Lead 调用 GitHub API │   │ 保持待命状态         │
│ 提交代码             │   │ 根据指示修正         │
│ (Token: $CCGHTK)    │   │                     │
└─────────────────────┘   └─────────────────────┘
```

### 3.3 Git 提交规范

使用环境变量 `CCGHTK` 进行 Token 鉴权：

```bash
# 示例提交命令
git push https://oauth2:${CCGHTK}@github.com/owner/repo.git main
```

## 4. 工作流程

### 4.1 标准迭代流程

```
用户需求 → Product Manager (PRD) → Developer (实现) → User Advocate (UAT) → Lead (验收) → 用户签收
    ↑                                                           │
    └─────────────────── 问题反馈 ←─────────────────────────────┘
```

### 4.2 沟通矩阵

| From \ To | Lead | Product Manager | Developer | User Advocate |
|-----------|------|-----------------|-----------|---------------|
| **Lead** | - | 需求澄清 | 资源调度 | 验收指示 |
| **Product Manager** | 风险上报 | - | PRD 交付 | 用例设计协作 |
| **Developer** | 技术风险 | 技术方案审核 | - | 测试环境准备 |
| **User Advocate** | 质量问题 | PRD 澄清 | Bug 报告 | - |
