## 2. 向量数据库选型

### 2.1 候选方案对比

| 维度 | ChromaDB | Qdrant | Milvus |
|------|----------|--------|--------|
| **部署复杂度** | ⭐⭐⭐⭐⭐ 极简 | ⭐⭐⭐⭐ 简单 | ⭐⭐⭐ 中等 |
| **性能** | 中等 | 高 | 极高 |
| **扩展性** | 有限 | 良好 | 优秀 |
| **内存占用** | 低 | 中 | 高 |
| **Python 集成** | 原生 | 良好 | 良好 |
| **持久化** | SQLite/duckDB | RocksDB | 多种后端 |
| **混合检索** | ❌ | ✅ | ✅ |
| **元数据过滤** | ✅ | ✅ | ✅ |

### 2.2 详细分析

#### ChromaDB
- **优势**: 开箱即用，无需额外服务，与 LangChain 深度集成
- **劣势**: 生产环境扩展性有限，不支持混合检索
- **适用场景**: 快速原型开发、小型项目

#### Qdrant
- **优势**: 性能与复杂度平衡，支持混合检索，Rust 实现高效
- **劣势**: 需要单独部署服务
- **适用场景**: 中型生产项目

#### Milvus
- **优势**: 企业级扩展性，支持十亿级向量
- **劣势**: 运维复杂度高，资源消耗大
- **适用场景**: 大规模企业部署

### 2.3 选型结论

**推荐方案: ChromaDB**

**理由**:
1. **快速落地**: Agent Builder 当前为单机部署，ChromaDB 无需额外服务
2. **低资源占用**: 使用 SQLite 持久化，对现有系统影响最小
3. **LangChain 集成**: Agent Builder 已使用 LangChain，集成成本低
4. **迁移路径**: 未来可通过 Qdrant/Milvus 进行升级

**备选方案**: 如需混合检索，可升级为 Qdrant

## 3. 文档解析引擎选型

### 3.1 候选方案对比

| 维度 | Marker | Unstructured | PyMuPDF4LLM |
|------|--------|--------------|-------------|
| **输出质量** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **处理速度** | 中等(需下载模型) | 快 | 极快 |
| **格式支持** | PDF/DOCX/PPTX/HTML | 30+ 格式 | 仅 PDF |
| **表格支持** | 优秀 | 良好 | 一般 |
| **模型依赖** | 1GB 模型 | 可选 | 无 |
| **LangChain 集成** | ❌ | ✅ | ✅ |

### 3.2 详细分析

#### Marker
- **优势**: 输出质量最高，保持原始布局，支持表格
- **劣势**: 首次使用需下载 1GB 模型，处理速度较慢
- **适用场景**: 对输出质量要求高的场景

#### Unstructured
- **优势**: 格式支持最广，LangChain 原生集成，模块化设计
- **劣势**: 高级功能需付费 API
- **适用场景**: 通用文档处理

#### PyMuPDF4LLM
- **优势**: 速度最快，无外部依赖
- **劣势**: 仅支持 PDF，复杂表格处理一般
- **适用场景**: 快速 PDF 处理

### 3.3 选型结论

**推荐方案: Unstructured（开源模式）+ PyMuPDF4LLM（备选）**

**理由**:
1. **LangChain 集成**: 与现有技术栈一致
2. **格式支持**: 原生支持 PDF 和 DOCX
3. **模块化**: 可按需启用不同解析器
4. **备选方案**: 遇到复杂 PDF 时回退到 PyMuPDF4LLM

**依赖安装**:
```bash
pip install unstructured[pdf,docx]
pip install PyMuPDF4LLM  # 备选
```

## 4. Embedding 模型选型

### 4.1 候选方案对比

| 方案 | 模型 | 维度 | 性能 | 成本 |
|------|------|------|------|------|
| ZhipuAI | embedding-3 | 2048 | 高 | API 计费 |
| OpenAI | text-embedding-3-small | 1536 | 高 | API 计费 |
| 本地模型 | bge-large-zh | 1024 | 中 | 免费 |
| 本地模型 | m3e-base | 768 | 中 | 免费 |

### 4.2 选型结论

**推荐方案: 复用现有 ModelServiceRegistry 的 Embedding 能力**

**理由**:
1. **统一管理**: 复用现有模型服务配置
2. **成本优化**: 使用已配置的 ZhipuAI/阿里百炼 API
3. **简化实现**: 无需额外部署本地模型

**实现方式**:
```python
# 通过 ModelServiceRegistry 获取 embedding
from src.model_service_registry import model_service_registry

embedding_service = model_service_registry.get_service("zhipu")
embeddings = await embedding_service.embed_documents(chunks)
```

## 5. 分块策略选型

### 5.1 候选策略

| 策略 | 描述 | 优势 | 劣势 |
|------|------|------|------|
| 固定大小 | 按 token 数分割 | 简单可控 | 可能切断语义 |
| 递归分割 | 按段落→句子→词递归 | 保持语义 | 参数敏感 |
| 语义分割 | 使用模型判断边界 | 最佳语义 | 计算成本高 |
| 文档结构 | 按标题/章节分割 | 保持结构 | 依赖文档格式 |

### 5.2 选型结论

**推荐方案: 递归字符分割 + 文档结构感知**

**理由**:
1. **LangChain 原生支持**: `RecursiveCharacterTextSplitter`
2. **语义保持**: 优先按段落分割
3. **格式适配**: 针对不同文档类型使用不同分隔符

**配置参数**:
```python
CHUNK_SIZE = 512  # 块大小
CHUNK_OVERLAP = 50  # 重叠大小
SEPARATORS = ["\n\n", "\n", "。", "！", "？", "；", " ", ""]  # 中文优先
```
