## 1. 需求背景

### 1.1 用户反馈

用户在使用"test"智能体进行调试对话时发现，当智能体已配置相关MCP工具时，LLM仍会选择直接回答而非调用工具：

| 用户输入 | 预期行为 | 实际行为 |
|---------|---------|---------|
| "讲一个冷笑话" | 调用 cold-jokes MCP | LLM直接编造笑话 |
| "3294/919+213" | 调用 calculator MCP | LLM直接计算 |
| "BTC的最新价格" | 调用 coingecko MCP | LLM直接回答（可能是过时数据） |

### 1.2 用户期望

当智能体已配置相关MCP工具时：
1. **优先使用工具**：有适用的工具时，必须调用工具
2. **LLM兜底**：仅当工具不适用或不可用时，才由LLM直接回答
3. **行为一致性**：工具调用决策应可预测、可解释

## 2. 问题根因分析

### 2.1 技术架构分析

当前系统的工具调用机制（`src/agent_engine.py:950-1030`）采用**纯文本协议**：

```python
# 当前实现：通过系统提示词"指令"LLM输出JSON
system_prompt += """
## 工具调用格式
调用工具时，**必须**严格按照以下 JSON 格式输出：
```json
{"tool": "工具名", "arguments": {}}
```
"""
```

**关键发现**：
- ❌ 未使用 LangChain 的 `bind_tools()` 方法
- ❌ 未使用 OpenAI Function Calling 的 `tools` 参数
- ✅ 仅依赖系统提示词中的"强制规则"文本

### 2.2 本质矛盾

| 层面 | 矛盾描述 |
|------|---------|
| **协议层** | 纯文本协议 vs 标准函数调用协议 |
| **约束力** | 软约束（提示词指令） vs 硬约束（API绑定） |
| **LLM行为** | LLM自主决定是否遵守指令 vs 被强制调用工具 |
| **可预测性** | 不确定（模型可能"抢答"） vs 确定（工具调用由API控制） |

### 2.3 为什么LLM会"抢答"？

根据业界最佳实践分析（见参考资料）：

1. **认知偏差**：LLM认为直接回答比调用工具更快（对简单任务如"讲笑话"）
2. **置信度陷阱**：LLM对自己的知识有信心，认为不需要外部工具
3. **指令遵守度**：不同模型对系统提示词的遵守程度差异巨大
4. **无强制约束**：当前纯文本协议无法强制LLM调用工具

### 2.4 业界最佳实践参考

根据搜索的业界资料：

| 来源 | 核心观点 |
|------|---------|
| [Anthropic - Effective Context Engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents) | 强调使用工具绑定（bind_tools）而非仅依赖提示词 |
| [LangChain - Tool-Calling Agent](https://medium.com/@dzianisv/vibe-engineering-langchains-tool-calling-agent-vs-react-agent-and-modern-llm-agent-architectures-bdd480347692) | 对比了ReAct与Tool-Calling Agent，后者通过API绑定工具 |
| [NetFoundry - AI Agent Tool Decision](https://netfoundry.io/ai/how-an-ai-agent-decides-to-call-mcp-tools/) | 分析了LLM决策工具调用的因素，强调工具定义的重要性 |

## 3. 需求规格

### 3.1 功能需求

#### FR-1: 工具调用优先级控制

**描述**：智能体必须优先使用已配置的工具，而非LLM直接回答。

**实现方案**：

1. **采用标准函数调用协议**
   - 使用 `llm.bind_tools(tool_schemas)` 绑定工具
   - 利用模型原生的 function calling 能力

2. **工具描述优化**
   - 增强 tool description，明确工具适用场景
   - 添加工具调用示例到 description

3. **系统提示词辅助**
   - 保留"优先使用工具"的指令
   - 添加"不要编造答案"的警告

#### FR-2: 工具适用性检测

**描述**：系统应帮助LLM准确判断何时应该调用工具。

**实现方案**：

1. **工具描述增强**
   ```python
   # 当前
   tool.description = "获取冷笑话"

   # 优化后
   tool.description = """获取冷笑话。当用户请求笑话、幽默内容时使用。
   - 适用场景：用户说"讲个笑话"、"来个冷笑话"
   - 不适用场景：用户询问笑话的原理/分类（知识性问题）
   - 返回：一个简短的笑话文本
   """
   ```

2. **工具元数据标签**
   - 添加 `keywords`: ["笑话", "joke", "幽默", "搞笑"]
   - 添加 `trigger_phrases`: ["讲", "来个", "来点"]

#### FR-3: 可观测性增强

**描述**：用户应能看到工具调用决策过程。

**实现方案**：

1. **思维链展示**
   - 显示"我正在分析是否需要使用工具..."
   - 显示"匹配到工具：calculator"
   - 显示"工具不适用，使用LLM直接回答"

2. **工具调用日志**
   - 记录每次工具调用的触发原因
   - 记录跳过工具调用的原因

### 3.2 非功能需求

| ID | 描述 | 优先级 |
|----|------|-------|
| NFR-1 | 工具调用不应显著增加首token时延 | P0 |
| NFR-2 | 向后兼容：不破坏现有的流式输出 | P0 |
| NFR-3 | 支持多模型提供商（Zhipu、Ollama等） | P1 |

## 4. 技术方案

### 4.1 架构变更

```
当前架构:
┌─────────────┐    System Prompt    ┌─────────────┐
│   Agent     │ ──────────────────> │     LLM     │
│   Engine    │ <────────────────── │             │
└─────────────┘   JSON Text Output  └─────────────┘

目标架构:
┌─────────────┐   bind_tools()      ┌─────────────┐
│   Agent     │ ──────────────────> │     LLM     │
│   Engine    │                     │ (with tools)│
└─────────────┘ <────────────────── └─────────────┘
              Tool Call/Result
```

### 4.2 关键代码变更

```python
# src/agent_engine.py

# 当前实现（纯文本协议）
async for chunk in self.llm.astream(messages):
    # 解析JSON格式工具调用

# 目标实现（标准函数调用）
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool

# 1. 定义工具Schema
tool_schemas = [convert_mcp_to_langchain_tool(t) for t in mcp_tools]

# 2. 绑定工具到LLM
llm_with_tools = self.llm.bind_tools(tool_schemas)

# 3. 流式调用并处理tool_calls
async for chunk in llm_with_tools.astream(messages):
    if hasattr(chunk, 'tool_calls') and chunk.tool_calls:
        # 处理标准tool_calls
```

### 4.3 兼容性处理

| 场景 | 处理方案 |
|------|---------|
| 模型不支持function calling | 回退到纯文本协议 |
| 已有对话历史 | 保持兼容，渐进式迁移 |
| Skill工具 | 保持现有机制（已是包装函数） |

## 6. 风险与依赖

### 6.1 风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 某些模型不支持function calling | 部分用户无法使用 | 保留纯文本协议作为fallback |
| bind_tools可能增加延迟 | 用户体验下降 | 性能测试，优化tool schemas |
| 破坏现有流式输出 | 核心功能失效 | 充分回归测试 |

### 6.2 依赖

- LangChain版本需支持 `bind_tools()` (≥0.1.0)
- 底层模型提供商需支持 function calling
