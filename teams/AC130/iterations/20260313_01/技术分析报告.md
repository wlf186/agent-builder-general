## 2. 问题根因分析

### 2.1 核心问题：提示词依赖型工具调用

| 特性 | 当前实现 | LangChain 原生工具调用 |
|------|----------|----------------------|
| 工具定义方式 | 文本描述 | 结构化 Tool 对象 |
| 工具绑定 | 无（仅提示词） | `bind_tools()` |
| LLM 决策方式 | 完全自主 | 模型原生支持 |
| 强制调用能力 | ❌ 无法强制 | ✅ `tool_choice="required"` |
| 解析方式 | 文本正则/JSON | 结构化响应 |

### 2.2 LLM 抢答的五个层次原因

#### Level 1: LLM 自主决策权
- 系统提示词只是"建议"，LLM 有权忽略
- 当 LLM 对答案有高置信度时，倾向于直接回答

#### Level 2: 缺乏结构化工具绑定
```python
# 当前：工具只是文本
system_prompt += "- evaluate: 计算数学表达式"

# 应该：结构化工具定义
tools = [Tool(name="evaluate", description="...", func=...)]
llm = llm.bind_tools(tools)  # ← 缺失这一步
```

#### Level 3: 无强制调用机制
```python
# 当前无法实现：
llm.bind_tools(tools).bind(tool_choice="any")  # 强制必须调用工具
```

#### Level 4: 文本解析的脆弱性
- 依赖 LLM 输出特定 JSON 格式
- LLM 可能输出：`"让我来计算：100+200*3 = 700"` 而非工具调用 JSON
- 检测逻辑：`if '"tool"' in buffer_content` 可能被绕过

#### Level 5: 模型能力差异
- 不同模型遵循指令的能力不同
- GLM-4.5-Air 可能比 GPT-4 更倾向于直接回答
- 温度参数影响输出稳定性

## 3. 修复方案建议

### 方案 A: 最小改动 - 增强提示词（短期）

**原理**: 通过更强的提示词工程提高工具调用率

**实施**:
```python
system_prompt += """

## ⚠️ 关键约束
1. 用户问的任何问题，如果涉及计算、查询、技能操作，你**没有能力直接回答**
2. 你必须先调用工具获取结果，然后基于工具结果回答
3. 直接回答任何需要工具支持的问题都是**错误行为**

## 检查清单（在回答前必须检查）
- [ ] 这个问题需要计算吗？ → 调用 evaluate
- [ ] 这个问题需要笑话？ → 调用 get_joke
- [ ] 这个问题需要技能？ → 调用 load_skill

如果以上任何一项为"是"，**禁止直接回答**！
"""
```

**优点**:
- 改动最小，风险低
- 无需修改架构

**缺点**:
- 仍然依赖 LLM 遵循指令
- 对某些模型效果有限

### 方案 B: 中等改动 - 添加 Tool Schema（中期）

**原理**: 使用 LangChain 的 `with_structured_output` 强制结构化输出

**实施**:
```python
from langchain_core.pydantic_v1 import BaseModel, Field
from langchain_core.output_parsers import JsonOutputParser

class ToolCall(BaseModel):
    """工具调用格式"""
    tool: str = Field(description="要调用的工具名称")
    arguments: dict = Field(description="工具参数")

# 创建结构化输出链
structured_llm = self.llm.with_structured_output(ToolCall)

# 使用时：
response = await structured_llm.ainvoke(messages)
if response.tool:
    # 执行工具调用
    result = await self._execute_tool(response.tool, response.arguments)
```

**优点**:
- 强制 LLM 输出结构化格式
- 提高工具调用检测准确率

**缺点**:
- 需要较大改动
- 可能影响流式输出

### 方案 C: 完整重构 - 原生 Tool Binding（长期）

**原理**: 使用 LangChain/LangGraph 的原生工具绑定机制

**实施**:
```python
from langchain_core.tools import StructuredTool
from langgraph.prebuilt import ToolNode

# 1. 定义工具（结构化）
tools = [
    StructuredTool.from_function(
        name="evaluate",
        description="计算数学表达式",
        func=lambda expr: str(eval(expr)),
        args_schema=EvaluateInput
    ),
    # ... 其他工具
]

# 2. 绑定到 LLM
llm_with_tools = self.llm.bind_tools(tools)

# 3. 使用 LangGraph 的 ToolNode
tool_node = ToolNode(tools)

# 4. 构建图
workflow = StateGraph(AgentState)
workflow.add_node("agent", agent_node)
workflow.add_node("tools", tool_node)
workflow.add_conditional_edges("agent", should_continue)
```

**优点**:
- 完整的原生支持
- 可以使用 `tool_choice="required"` 强制调用
- 与 LangGraph 生态完全兼容

**缺点**:
- 需要重构大量代码
- MCP 工具需要适配为 LangChain Tool 格式
- 流式输出需要重新设计

### 方案 D: 混合方案 - 双层检测（推荐）

**原理**: 结合提示词增强 + 后处理检测

**实施**:
```python
# 1. 增强提示词
system_prompt += """
## 强制工具调用检查
在给出任何答案前，你必须自检：
1. 这个问题是否在我的能力范围内？
2. 我是否需要调用工具来获取正确答案？

如果答案不确定，必须调用工具！
"""

# 2. 添加响应后检测
async def _check_and_correct_response(self, user_input: str, response: str):
    """检测并纠正抢答行为"""

    # 检测关键词
    trigger_keywords = ["计算", "多少", "求", "等于", "笑话", "pdf", "docx"]
    needs_tool = any(kw in user_input.lower() for kw in trigger_keywords)

    if needs_tool and not self._has_tool_call(response):
        # 检测到抢答，重新生成
        yield {"type": "thinking", "content": "⚠️ 检测到需要工具调用，重新处理..."}
        messages.append(SystemMessage(content="请使用工具回答这个问题"))
        # 重新调用 LLM
        return await self._regenerate_with_tool_forcing(messages)

    return response
```

**优点**:
- 兼顾现有架构
- 可配置的检测规则
- 渐进式改进

## 5. 总结

### 根因
LLM 抢答的根本原因是：**当前系统使用"提示词依赖型"工具调用机制，完全依赖 LLM 自主决定是否使用工具，缺乏结构化工具绑定和强制调用机制。**

### 关键证据
1. 无 `bind_tools()` 调用
2. 无 `tool_choice` 配置
3. 工具仅作为文本描述嵌入提示词
4. 工具调用依赖文本解析而非结构化响应

### 优先级
| 方案 | 实施难度 | 预期效果 | 优先级 |
|------|----------|----------|--------|
| A - 提示词增强 | 低 | 中 | 高 |
| D - 混合检测 | 中 | 高 | 中 |
| B - 结构化输出 | 高 | 高 | 中 |
| C - 原生绑定 | 极高 | 极高 | 低 |
