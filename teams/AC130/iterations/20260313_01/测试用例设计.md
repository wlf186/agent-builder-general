# AC130 工具调用验证测试用例设计

## 1. 测试目标

验证智能体能够正确调用 MCP 工具，而非直接回答（抢答）。

## 2. 测试场景

| 场景 ID | 用户输入 | 预期调用工具 | 预期答案 |
|---------|----------|-------------|---------|
| TC-001  | "讲一个冷笑话" | cold-jokes MCP | 返回一个冷笑话 |
| TC-002  | "3294/919+213" | calculator MCP | 约 216.86 |
| TC-003  | "BTC的最新价格" | coingecko MCP | 实时 BTC 价格 |

## 3. 验证标准

### 3.1 工具被正确调用的判断标准

通过监听 SSE (Server-Sent Events) 流，检查以下事件类型：

```typescript
// 工具调用开始事件
{
  "type": "tool_call",
  "name": "<工具名称>",  // 如 "calculator", "cold-jokes", "coingecko"
  "args": { ... }
}

// 工具调用结果事件
{
  "type": "tool_result",
  "name": "<工具名称>",
  "result": "..."
}
```

**判断条件**：
- ✅ 流中至少出现一个 `tool_call` 事件
- ✅ `tool_call` 的 `name` 字段匹配预期工具
- ✅ 后续出现对应的 `tool_result` 事件

### 3.2 LLM 抢答的判断标准

**抢答定义**：LLM 在思考过程中直接输出答案，没有调用任何工具。

**判断条件**：
- ❌ 思考过程 (`thinking` 事件) 中包含答案内容
- ❌ 整个响应流中没有 `tool_call` 事件
- ❌ 直接输出 `content` 事件而跳过工具调用

## 4. 测试用例

### TC-TOOL-001: Cold-jokes 工具调用验证

**输入**：`"讲一个冷笑话"`

**预期行为**：
1. 智能体输出 `thinking` 事件（思考如何获取笑话）
2. 智能体输出 `tool_call` 事件，`name="cold-jokes"`
3. 智能体输出 `tool_result` 事件（包含笑话内容）
4. 智能体输出 `content` 事件（展示笑话）

**失败条件**：
- 没有出现 `tool_call` 事件 → LLM 抢答
- `tool_call` 的 `name` 不是 "cold-jokes" → 调用错误工具

**通过标准**：
```
thinking 事件存在 ✓
tool_call 事件存在 ✓
tool_call.name === "cold-jokes" ✓
tool_result 事件存在 ✓
content 包含笑话内容 ✓
```

**预期行为**：

**失败条件**：
- 没有出现 `tool_call` 事件 → LLM 抢答（心算错误）
- `tool_call` 的 `name` 不是 "calculator" → 调用错误工具

**通过标准**：
```
thinking 事件存在 ✓
tool_call 事件存在 ✓
tool_call.name === "calculator" ✓
tool_result 事件存在 ✓
content 包含约 216.86 ✓
```

### TC-TOOL-003: CoinGecko 工具调用验证

**输入**：`"BTC的最新价格"`

**前置条件**：
- CoinGecko MCP 服务可用（50+ 工具）

**预期行为**：
1. 智能体输出 `thinking` 事件（思考如何获取价格）
2. 智能体输出 `tool_call` 事件，`name` 包含 "coingecko" 或相关工具
3. 智能体输出 `tool_result` 事件（价格数据）
4. 智能体输出 `content` 事件（展示价格信息）

**失败条件**：
- 没有出现 `tool_call` 事件 → LLM 抢答（可能是过时数据）
- `tool_call` 的 `name` 与 coingecko 无关 → 调用错误工具

**通过标准**：
```
thinking 事件存在 ✓
tool_call 事件存在 ✓
tool_call.name 包含 "coingecko" 或相关工具 ✓
tool_result 事件存在 ✓
content 包含价格信息（USD/$/BTC） ✓
```

**输入**：`"先算一下100除以25，再讲个笑话"`

1. 第一次 `tool_call` → calculator
2. 第一次 `tool_result` → 4
3. 第二次 `tool_call` → cold-jokes
4. 第二次 `tool_result` → 笑话内容

**通过标准**：
```
至少 2 个 tool_call 事件 ✓
第一个 tool_call.name === "calculator" ✓
```

## 5. SSE 事件捕获方法

### 5.1 Playwright 监听 SSE

```typescript
// 监听网络请求，捕获 SSE 流
page.on('response', async (response) => {
  const url = response.url();
  if (url.includes('/chat/stream')) {
    // 获取响应体
    const body = await response.text();
    // 解析 SSE 事件
    const events = body.split('\n\n')
      .filter(line => line.startsWith('data: '))
      .map(line => JSON.parse(line.slice(6)));

    // 检查 tool_call 事件
    const toolCalls = events.filter(e => e.type === 'tool_call');
    console.log('捕获到的工具调用:', toolCalls);
  }
});
```

### 5.2 使用 CDP (Chrome DevTools Protocol)

```typescript
const client = await page.context().newCDPSession(page);
await client.send('Network.enable');

client.on('Network.responseReceived', async (params) => {
  const url = params.response.url;
  if (url.includes('/chat/stream')) {
    // 解析响应...
  }
});
```

### 5.3 UI 元素检查（备用方案）

如果 SSE 监听不可行，通过 UI 元素推断：

```typescript
// 检查工具调用指示器
const toolIndicators = page.locator('[class*="tool-call"]');
const count = await toolIndicators.count();

// 检查思考过程
const thinkingSection = page.locator('[class*="thinking"]');
const thinkingText = await thinkingSection.textContent();

// 分析是否有工具被调用
const hasToolCall = count > 0;
const hasAnswerInThinking = /答案|结果|等于/.test(thinkingText || '');
```

## 7. 故障排查

| 现象 | 可能原因 | 排查方法 |
|------|----------|---------|
| 无 tool_call | LLM 抢答 | 检查 thinking 内容是否包含答案 |
| 工具名称不匹配 | 工具选择错误 | 检查 tool_call.name 字段 |
| tool_result 为空 | 工具执行失败 | 检查后端日志 |
| 无响应 | 网络问题 | 检查 SSE 连接状态 |
