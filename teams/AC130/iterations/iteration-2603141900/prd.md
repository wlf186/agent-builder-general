## 1. 背景与目标

### 1.1 业务背景

Agent Builder 系统集成了多个 MCP（Model Context Protocol）服务和 Skill 工具，包括：
- **Calculator MCP**: 数学计算工具（evaluate、add、subtract、multiply、divide、power、sqrt）
- **Cold Jokes MCP**: 冷笑话生成工具（get_joke、list_categories、get_jokes_by_category）
- **CoinGecko MCP**: 加密货币数据查询（50+ 工具）
- **PDF/DOCX Skills**: 文件处理技能

**用户反馈问题**：

用户报告智能体在以下场景中**未调用 MCP/Skill 工具**，而是使用 LLM 自身的知识直接回答：

| 用户请求 | 期望行为 | 实际行为 | 问题 |
|---------|----------|----------|------|
| "计算 100+200*3" | 调用 evaluate 工具 | LLM 直接计算结果 | 结果可能不准确，未验证工具能力 |
| "讲个冷笑话" | 调用 get_joke 工具 | LLM 生成一个笑话 | 笑话质量不稳定，未使用预置内容 |
| "比特币现在的价格" | 调用 CoinGecko 工具 | LLM 可能回答过时数据 | 数据时效性无保障 |

### 1.2 问题根因分析

经过代码审查（`src/agent_engine.py`），发现以下问题：

**1. 系统提示词虽有强制规则，但 LLM 可能不遵守**

```python
# 当前系统提示词 (agent_engine.py:1103-1115)
## 🔴 强制规则：必须使用工具
**重要：你没有任何内置计算能力！** 所有计算、获取笑话、加载技能等操作，**必须**通过调用工具完成。

- ❌ 禁止：自己计算数学表达式
- ✅ 必须：调用 evaluate 工具计算任何数学表达式
- ❌ 禁止：自己编造笑话
- ✅ 必须：调用 get_joke 工具获取笑话
```

**问题**：LLM 模型（尤其是开源模型如 Qwen、GLM）对系统提示词的遵循度参差不齐，当任务简单时可能忽略规则直接回答。

**2. 工具绑定正确，但缺少优先级策略**

```python
# 当前工具绑定 (agent_engine.py:166-207)
def _bind_tools_to_llm(self):
    tools_to_bind = []
    # 1. MCP 工具
    if self.mcp_manager and self.mcp_manager.all_tools:
        mcp_tools = self._tool_adapter.convert_all_tools()
        tools_to_bind.extend(mcp_tools)
    # 2. Skill 工具
    if self.skill_tool and self.skill_tool.enabled_skills:
        skill_tools = self._create_skill_tools()
        tools_to_bind.extend(skill_tools)
```

**问题**：工具已正确绑定到 LLM，但当 LLM 使用原生工具调用（`bind_tools`）时，它仍可能选择不调用工具而直接生成文本回答。

**3. 工具描述质量参差不齐**

| 工具 | 当前描述 | 问题 |
|------|----------|------|
| `evaluate` | "计算数学表达式的值。支持加减乘除、幂运算、括号等..." | 描述清晰，但缺少"优先使用"的强调 |
| `get_joke` | "获取一个随机冷笑话，让人开心一下" | 过于通用，未强调"不要自己编造" |

### 1.3 目标

本次迭代旨在**提升 MCP/Skill 工具的调用优先级和准确性**：

1. **工具优先级策略**：明确工具调用优先级，当有匹配工具时优先使用
2. **工具描述优化**：增强工具描述，强调"必须使用"的语义
3. **兜底逻辑**：当工具调用失败时提供明确的降级方案
4. **流式输出兼容**：所有优化不得影响现有的流式输出体验

### 1.4 成功指标

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 数学计算工具调用率 | > 95% | 日志分析：计算类请求的工具调用比例 |
| 冷笑话工具调用率 | > 90% | 日志分析：笑话类请求的工具调用比例 |
| CoinGecko 工具调用率 | > 85% | 日志分析：加密货币查询的工具调用比例 |
| 流式输出首 token 时延 | < 500ms | 性能监控 |
| 流式输出打字机效果 | 正常 | 人工验证 |

#### F-001: 工具优先级策略配置

**用户故事**：
```
作为系统管理员
我希望可以配置工具调用优先级策略
以便确保智能体优先使用 MCP/Skill 工具而非 LLM 直接回答
```

**验收标准**：
- [ ] Given 配置了 `tool_call_strategy="required"` When 智能体处理任何请求 Then 必须调用至少一个工具
- [ ] Given 配置了 `tool_call_strategy="auto"` When 智能体处理请求 Then LLM 自主决定是否调用工具
- [ ] Given 用户请求与工具功能匹配 When 评估是否调用工具 Then 优先匹配的工具被调用

**策略类型**：

| 策略 | 值 | 说明 | 适用场景 |
|------|-----|------|----------|
| **强制调用** | `required` | 必须调用工具，否则报错 | 确保工具被使用（如测试环境） |
| **优先调用** | `preferred` | 有匹配工具时优先使用，否则降级 | 生产环境推荐 |
| **自动判断** | `auto` | LLM 自主决定 | 兼容模式 |

**技术实现**：
```python
class ToolCallStrategy(str, Enum):
    REQUIRED = "required"   # 强制调用（OpenAI 兼容）
    PREFERRED = "preferred" # 优先调用（新增）
    AUTO = "auto"           # 自动判断（默认）
```

#### F-002: 工具描述优化

**用户故事**：
```
作为智能体
我希望工具描述能明确告诉我何时以及如何使用工具
以便我能够正确地调用工具而不是自己编造答案
```

**验收标准**：
- [ ] Given 工具描述 When 生成 Then 包含"必须使用"、"禁止"等强语义词
- [ ] Given 工具描述 When 生成 Then 包含至少一个调用示例
- [ ] Given 用户请求与工具匹配 When 工具描述 Then 工具名称和功能高度相关

**工具描述模板**：

```json
{
  "name": "evaluate",
  "description": "🔴 【必须使用】计算数学表达式的值。当用户请求任何数学计算时，必须调用此工具，禁止自行计算！支持加减乘除、幂运算、括号、小数等复杂表达式。例如：evaluate('100+200*3/4') 返回计算结果。",
  "inputSchema": {...}
}

{
  "name": "get_joke",
  "description": "🔴 【必须使用】获取一个冷笑话。当用户请求笑话、幽默内容时，必须调用此工具，禁止自行编造笑话！支持按类别获取：animal(动物)、food(食物)、tech(科技)、daily(日常)。",
  "inputSchema": {...}
}

{
  "name": "get_coin_price",
  "description": "🔴 【必须使用】查询加密货币价格。当用户询问比特币、以太坊等加密货币的价格或市场数据时，必须调用此工具，禁止使用过时的训练数据回答！",
  "inputSchema": {...}
}
```

**描述优化原则**：
1. **强语义前缀**：使用 "🔴 【必须使用】" 等视觉标记
2. **明确触发条件**：描述"当用户请求 X 时，必须调用此工具"
3. **禁止行为说明**：明确"禁止自行 Y"
4. **示例引导**：提供具体的调用示例

#### F-003: 工具匹配规则引擎

**用户故事**：
```
作为系统
我希望能够在用户请求与工具之间建立精确的匹配关系
以便在合适的时候触发合适的工具
```

**验收标准**：
- [ ] Given 用户请求包含"计算"、"加"、"减"、"乘"、"除"等关键词 When 匹配 Then 优先调用 calculator 工具
- [ ] Given 用户请求包含"笑话"、"幽默"、"开心"等关键词 When 匹配 Then 优先调用 cold-jokes 工具
- [ ] Given 用户请求包含"比特币"、"BTC"、"价格"等关键词 When 匹配 Then 优先调用 coingecko 工具

**匹配规则配置**：

```python
# src/tool_matcher.py
TOOL_MATCHING_RULES = {
    "calculator": {
        "keywords": ["计算", "加", "减", "乘", "除", "等于", "数学", "求", "eval"],

        "priority": 10,
        "tool_names": ["evaluate", "add", "subtract", "multiply", "divide", "power", "sqrt"]
    },
    "cold-jokes": {
        "keywords": ["笑话", "幽默", "开心", "讲个", "逗", "冷笑话"],
        "priority": 8,
        "tool_names": ["get_joke", "list_categories"]
    },
    "coingecko": {
        "keywords": ["比特币", "BTC", "以太坊", "ETH", "加密货币", "币价", "价格"],
        "priority": 9,
        "tool_names": ["get_coin_price", "get_market_data"]
    }
}
```

**匹配流程**：
```
┌─────────────────────────────────────────────────────────────┐
│ 用户输入: "计算 100+200*3"                                   │

                                 ▼
```

#### F-004: 工具调用监控与诊断

**用户故事**：
```
作为开发者
我希望能够监控工具调用的详细情况
以便诊断为什么智能体没有调用某个工具
```

**验收标准**：
- [ ] Given 智能体处理请求 When 完成响应 Then 记录工具调用情况
- [ ] Given 工具未被调用 When 用户请求与工具匹配 Then 记录未调用原因
- [ ] Given 用户查询诊断信息 When 请求诊断 API Then 返回详细的工具调用分析

**诊断数据结构**：

```python
@dataclass
class ToolCallDiagnostic:
    """工具调用诊断数据"""
    agent_name: str
    user_query: str
    timestamp: str

    # 工具匹配情况
    matched_tools: List[str]          # 匹配到的工具名称
    tool_scores: Dict[str, int]       # 各工具的匹配评分

    # 实际调用情况
    tools_called: List[str]           # 实际调用的工具
    tools_skipped: List[str]          # 跳过的工具

    # 未调用原因分析
    skip_reasons: Dict[str, str]      # 工具名 → 未调用原因

    # 建议
    recommendations: List[str]        # 改进建议
```

**诊断 API**：

```
GET /api/agents/{name}/diagnose?query=计算100+200

Response:
{
  "query": "计算100+200",
  "matched_tools": ["evaluate"],
  "tool_scores": {"evaluate": 30},
  "tools_called": [],
  "tools_skipped": ["evaluate"],
  "skip_reasons": {
    "evaluate": "LLM 选择直接回答而非调用工具。建议：检查工具描述，强化'必须使用'语义"
  },
  "recommendations": [
    "优化 evaluate 工具描述，增加'🔴 【必须使用】'前缀",
    "考虑设置 tool_call_strategy='preferred' 强制优先使用工具"
  ]
}
```

#### F-005: 兜底逻辑与降级方案

**用户故事**：
```
作为用户
我希望当工具调用失败时智能体能够优雅降级
以便获得合理的回答而不是错误信息
```

**验收标准**：
- [ ] Given 工具调用失败 When 可重试错误 Then 自动重试最多 3 次
- [ ] Given 工具调用失败 When 不可重试错误 Then 降级到 LLM 直接回答并说明
- [ ] Given 工具超时 When 等待 > 30 秒 Then 放弃工具调用，降级处理

**降级策略**：

| 场景 | 检测条件 | 降级行为 | 用户提示 |
|------|----------|----------|----------|
| 工具超时 | 等待 > 30 秒 | 放弃调用，使用 LLM 回答 | "工具响应超时，以下是我的估算..." |
| 工具错误 | HTTP 500/502/503 | 重试 3 次，仍失败则降级 | "工具暂时不可用，以下是我的回答..." |
| 工具不存在 | 404 Not Found | 降级到 LLM | "相关工具暂时离线..." |
| 网络错误 | 连接失败 | 降级到 LLM | "无法连接到工具服务..." |

### 5.3 限制条件

1. **LLM 模型差异**：不同模型对工具调用指令的遵循度不同
2. **网络依赖**：MCP 服务（如 CoinGecko）依赖网络连接
3. **向后兼容**：需要保持对现有配置的兼容性
