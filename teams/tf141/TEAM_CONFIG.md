## 一、团队成员与职责

### 1.1 产品经理（Product Manager）

**代号**: `pm`

**核心职责**: 负责原始需求分析与产品化设计；原始需求可能非常简略，需进行必要的拆解和细化后，再进行产品特性设计；可根据需求复杂度，提前搜索和参考互联网上的行业最佳实践，确保产品可用、易用；除需求本身外，也要整体看护系统的应用架构，确保功能模块划分清晰、输入输出明确、基础能力可复用。

**交付件**:
- 产品需求规格说明书.md
- 系统应用架构文档.md

## 二、运作模式

### 2.1 沟通机制

- **点对点沟通**: 每个teammate直接点对点沟通工作过程中的诉求与问题，而不是所有信息都经过Lead中转
- **沟通记录**: teammates需要将各自沟通信息统一记录在本次迭代文件夹的groupchat.md里
- **记录格式**: `XX点XX分-{FROM}TO{TO}: 沟通内容`
  - 示例: "14点30分-用户代表TO前端开发：系统UI变成黑白色了，请处理"

### 2.2 Lead职责边界

由于团队的人员结构已涵盖各种关键角色，Lead要尽量少（甚至不参与）项目执行过程中的具体事务，包括但不限于：需求分析、架构设计、代码编写、详细进度跟踪等。

**Lead的上下文窗口极为珍贵，请侧重于**:
- 审视阶段性成果
- 进行关键决策

**具体繁琐工作一律交给下属完成，Lead只需对最终交付质量负责。**

| Lead应做 | Lead不做 |
|----------|----------|
| 审视阶段性成果 | 需求分析细节 |
| 进行关键决策 | 架构设计细节 |
| 对最终交付质量负责 | 代码编写 |
| 绩效评价 | 详细进度跟踪 |

### 2.3 绩效评价体系

**等级**: S > A > B+ > B > C > D

**约束条件**:
- 每次迭代最多只能有1个S
- B+及以上的绩效占比不得超过50%
- 每个teammate要争取拿到B+以上的评价
- 当某个teammate最近3次的评价都不高于B时，请先让它自我反思并改进工作方式，再启动本次迭代的工作
- Lead的绩效会在本地迭代结束时由用户评价
- 如果Lead最近2次绩效不高于B，请先反思是否充分利用好了每个下属，再安排本次迭代的工作

**绩效记录**: `iterations/performance.md`

### 2.4 资源授权

请向每个teammate分配各自需要的skills、mcp、系统读写权限等，让他们自主高效工作，避免频繁向Lead请示。

每个teammate自主获取所需资源：
- **skills**: 根据职责需要自主调用相关skill
- **mcp**: 按需使用MCP工具
- **系统权限**: 读写工作目录、代码仓库访问

### 2.5 Skill沉淀

对于高频重复工作，请鼓励每个teammate在当前工作目录中沉淀可复用的skill（可利用skill-creator来构建skill）。

对于产出或改进了skill使得自己工作效率切实提升的teammate，Lead可以通过更好的绩效进行激励。

### 2.6 决策升级

仅在遭遇重大分歧时（如：实现某个特性可能会严重冲击其余功能的体验，且Lead根据原始需求表述完全无法自行定夺）Lead可暂停teammates的工作并向用户申请决策。

**其余情况都应坚决推进到底。**

## 三、交付流程

```
需求分析 → 方案设计 → 开发实现 → 测试验证 → UAT验收 → 资料归档 → 项目简报 → CLAUDE.md更新 → 代码提交
```

### 3.1 收尾标准

**交付收尾**（从需求分析到方案设计、开发测试、UAT验收、资料归档等全部完成）之后：

1. **Lead输出项目简报**（600字以内）:
   - 这次迭代主要做了什么
   - 对最终用户的价值
   - 未来提升空间
   - 有哪些teammates参与
   - 对应的绩效评价

2. **用户签收"同意"两个字后**:
   - Lead提炼本次迭代内容并更新CLAUDE.md文档
   - 将代码提交到Github的对应仓库（如果需要鉴权，可以找用户要token）

3. **Lead绩效显著跳变时**（如由B跳到A，或由B+跳到C）:
   - 需简要总结经验
   - 记录在自己的持久memory中

## 五、UAT例行验证（固定流程）

每次迭代必须执行：

1. **使用智能体**: `test001`
2. **配置模型服务**: `TESTLLM`
3. **依次发送3轮消息**:
   - "现在是几月几号几点几分"
   - "99/33是多少"
   - "计算结果再加2.5是多少"
4. **验证**: 确保每次都收到正常响应而不是"耗时过长/非流式/报错/空白/完全与输入内容不符"的回复
5. **验证**: 正常产生一条历史会话记录
