## 问题概述

本次迭代针对客户反馈的三个核心Bug进行修复：

1. **会话记录串台问题** - 不同智能体的会话记录互相可见
2. **Skill勾选与执行不一致** - 勾选的Skill与实际执行时的名称不匹配
3. **历史会话共享问题** - 同一智能体的历史会话需要跨用户共享

## 问题一：会话记录串台问题

### 1.1 问题描述

**现象**：不同智能体的会话记录互相可见，A智能体里输入的信息在B、C、D里都能看到，且历史会话也一样。

**预期**：每个智能体的配置信息、会话记录、Skill执行环境等应该是完全独立的。

### 1.2 根因分析

经过代码审查，发现后端会话存储逻辑是正确的：

```python
# src/conversation_manager.py
def _get_agent_dir(self, agent_name: str) -> Path:
    """获取智能体对应的会话目录"""
    agent_dir = self.conversations_dir / agent_name  # 按agent_name隔离
    agent_dir.mkdir(parents=True, exist_ok=True)
    return agent_dir
```

后端API也正确传递了agent_name参数：

```python
# backend.py
@app.get("/api/agents/{name}/conversations")
async def list_conversations(name: str):
    conversations = conversation_manager.list_conversations(name)  # 使用name隔离
```

**问题定位在前端**：

1. **状态管理问题**：`page.tsx` 中 `currentConversationId` 和 `currentConversationMessages` 是全局状态，切换智能体时未正确重置。

2. **缓存问题**：`ConversationDrawer` 组件的 `conversations` 状态在切换智能体后可能未及时刷新。

3. **初始化时机问题**：当用户切换到不同智能体时，历史会话列表未立即清空并重新加载。

**代码证据**：
```typescript
// frontend/src/app/page.tsx
const [currentConversationId, setCurrentConversationId] = useState<string | null>(null);
const [currentConversationMessages, setCurrentConversationMessages] = useState<any[]>([]);

// 切换智能体时（openAgentConfig函数）没有重置这些状态
const openAgentConfig = async (name: string) => {
  // ... 加载配置
  setSelectedAgent(name);
  // 缺失：setCurrentConversationId(null);
  // 缺失：setCurrentConversationMessages([]);
};
```

### 1.3 解决方案

#### 需求定义

| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| REQ-1.1 | 切换智能体时自动清空当前会话ID和消息列表 | P0 |
| REQ-1.2 | ConversationDrawer组件在agentName变化时自动重新加载会话列表 | P0 |
| REQ-1.3 | AgentChat组件在agentName变化时重置内部状态 | P0 |
| REQ-1.4 | 添加会话隔离的单元测试 | P1 |

#### 技术实现要点

1. **page.tsx 修改**：
   ```typescript
   const openAgentConfig = async (name: string) => {
     // 重置会话状态
     setCurrentConversationId(null);
     setCurrentConversationMessages([]);
     // ... 其余逻辑
   };
   ```

2. **AgentChat.tsx 修改**：
   ```typescript
   useEffect(() => {
     // agentName 变化时重置状态
     setMessages([]);
     setFileContext({ file_ids: [], file_infos: [] });
   }, [agentName]);
   ```

3. **ConversationDrawer.tsx 修改**：
   ```typescript
   useEffect(() => {
     if (open && agentName) {
       setConversations([]); // 先清空，避免显示旧数据
       loadConversations();
     }
   }, [open, agentName]);
   ```

### 1.4 验收标准

| 编号 | 验收条件 |
|------|----------|
| AC-1.1 | 切换智能体后，会话抽屉显示的是当前智能体的会话列表 |
| AC-1.2 | 在A智能体创建的会话不会出现在B智能体的会话列表中 |
| AC-1.3 | 切换智能体后，聊天区域自动清空，不显示之前的对话 |
| AC-1.4 | 数据目录 `data/conversations/` 下按智能体名称正确隔离存储 |

## 问题二：Skill勾选与执行不一致

### 2.1 问题描述

**现象**：用户勾选的是 "AB-DOCX creation, editing, and analysis" 这个skill，但智能体执行时报错 "AB-docx" 这个skill不存在。

**分析**：
- 勾选的显示名称：`AB-DOCX creation, editing, and analysis`（来自SKILL.md的标题）
- 实际注册的skill名称：`AB-docx`（来自SKILL.md的frontmatter或目录解析）
- LLM调用时使用的名称：可能是从标题解析而非注册名称

### 2.2 根因分析

**Skill名称来源有两个地方**：

1. **注册名称**（来自 `skill_registry.py`）：
   ```python
   # 从 SKILL.md 的 frontmatter 获取
   if not name and 'name' in metadata:
       name = metadata['name']  # AB-docx
   # 或从第一个标题获取
   title_match = re.search(r'^#\s+(.+)$', content, re.MULTILINE)
   if title_match:
       name = title_match.group(1).strip()  # AB-DOCX creation, editing, and analysis
   ```

2. **实际SKILL.md文件**：
   ```markdown
   ---
   name: AB-docx  # ← 注册时使用这个
   ---
   # AB-DOCX creation, editing, and analysis  # ← 或者这个
   ```

**问题在于**：
- 前端展示使用的是 `skill.name`（可能是标题）
- 后端注册时可能使用的是 frontmatter 中的 `name`
- LLM工具描述中列出的是 `skill.name`
- 但 `skill_tool.py` 的 `enabled_skills` 列表使用的是注册时的名称

**代码证据**：
```python
# src/skill_tool.py
def __init__(self, skill_registry, skills_dir, enabled_skills, ...):
    self.enabled_skills = enabled_skills  # 这是Agent配置中的skills列表

async def execute(self, skill_name: str, ...):
    if skill_name not in self.enabled_skills:  # 检查是否在启用列表中
        return f"Error: Skill '{skill_name}' is not available."
```

如果前端保存的 `config.skills` 列表使用的是标题名称（如 "AB-DOCX creation, editing, and analysis"），而后端注册表使用的是 "AB-docx"，就会导致匹配失败。

### 2.3 解决方案

#### 需求定义

| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| REQ-2.1 | 统一Skill名称规范，使用frontmatter中的name作为唯一标识 | P0 |
| REQ-2.2 | 前端展示Skill时同时显示ID和友好名称 | P1 |
| REQ-2.3 | Agent配置保存的skills列表使用Skill ID而非显示名称 | P0 |
| REQ-2.4 | 后端Skill注册时强制校验name字段存在性 | P0 |

#### 技术实现要点

1. **skill_registry.py 修改** - 统一名称来源：
   ```python
   def _parse_skill_md(self, skill_path: Path) -> Tuple[str, str, Dict[str, Any]]:
       # 优先使用 frontmatter 中的 name
       if 'name' in metadata:
           name = metadata['name']
       else:
           # 如果没有 frontmatter，从标题提取并规范化
           title_match = re.search(r'^#\s+(.+)$', content, re.MULTILINE)
           if title_match:
               # 只取第一个单词作为ID（如 "AB-DOCX creation..." -> "AB-DOCX"）
               name = title_match.group(1).strip().split()[0]

       # 名称规范化：统一转为小写
       normalized_name = name.lower().replace(" ", "-")
       return normalized_name, description, metadata
   ```

2. **models.py 修改** - 添加显示名称字段：
   ```python
   class SkillConfig(BaseModel):
       name: str           # 唯一标识符（如 "ab-docx"）
       display_name: str   # 显示名称（如 "AB-DOCX creation, editing, and analysis"）
       description: str
       # ...
   ```

3. **skill_tool.py 修改** - 增强名称匹配：
   ```python
   async def execute(self, skill_name: str, ...):
       # 尝试精确匹配
       if skill_name in self.enabled_skills:
           actual_name = skill_name
       else:
           # 尝试模糊匹配（大小写、前缀）
           actual_name = self._fuzzy_match_skill(skill_name)
           if not actual_name:
               return f"Error: Skill '{skill_name}' not found."

       # 使用匹配后的名称继续执行
       ...
   ```

4. **前端修改** - 保存时使用ID：
   ```typescript
   // 保存Agent配置时，确保使用skill.name（ID）而非display_name
   const skillsToSave = selectedSkills.map(s => s.name);
   ```

### 2.4 验收标准

| 编号 | 验收条件 |
|------|----------|
| AC-2.1 | 所有builtin skills的名称统一为小写连字符格式（如 "ab-docx", "ab-pdf"） |
| AC-2.2 | 前端Skill选择列表显示友好名称，但保存时使用ID |
| AC-2.3 | LLM调用load_skill时使用正确的Skill ID |
| AC-2.4 | 即使LLM使用了大小写错误的名称（如"AB-DOCX"），也能正确匹配到"ab-docx" |
| AC-2.5 | 技能加载失败时，错误信息中列出所有可用的Skill ID |

## 问题三：历史会话共享问题

### 3.1 问题描述

**现象**：用户代表对test001智能体进行了3轮例行测试，过一会真实用户登录系统时，应该在test001里也能看到这3轮测试的情况。

**当前行为**：会话记录可能只存在于当前浏览器会话中，不同用户/会话之间不共享。

**预期**：同一智能体的历史会话记录应该对所有用户可见（因为是智能体级别的数据，不是用户级别的数据）。

### 3.2 根因分析

**当前架构分析**：

1. **后端存储**：会话数据已正确存储在服务器端
   ```
   data/conversations/{agent_name}/{conversation_id}.json
   ```

2. **前端加载**：每次打开会话抽屉时都会调用API
   ```typescript
   useEffect(() => {
     if (open && agentName) {
       loadConversations();  // 调用 /api/agents/${agentName}/conversations
     }
   }, [open, agentName]);
   ```

**问题可能原因**：

1. **浏览器缓存**：前端可能缓存了会话列表，没有每次都重新请求
2. **会话状态未持久化**：用户A的对话可能只保存在前端状态，未及时同步到后端
3. **同步时机问题**：消息发送后可能未立即保存到后端的conversation记录中

**代码证据**：
```typescript
// AgentChat.tsx 中消息保存逻辑
// 需要确认每次对话结束后是否调用 saveConversationMessages API
```

检查发现：`AgentChat.tsx` 组件中流式对话完成后，消息需要显式调用后端API保存。如果用户在保存前刷新页面或关闭，数据可能丢失。

### 3.3 解决方案

#### 需求定义

| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| REQ-3.1 | 每次对话完成后立即保存到后端（不依赖手动保存） | P0 |
| REQ-3.2 | 会话列表加载时禁用浏览器缓存 | P0 |
| REQ-3.3 | 后端会话API添加ETag/Last-Modified支持，确保获取最新数据 | P1 |
| REQ-3.4 | 添加会话同步状态指示器，让用户知道数据是否已保存 | P2 |

#### 技术实现要点

1. **AgentChat.tsx 修改** - 自动保存：
   ```typescript
   // 在流式对话完成后自动保存
   const saveConversation = useCallback(async () => {
     if (!conversationId || messages.length === 0) return;

     try {
       await fetch(`/api/agents/${agentName}/conversations/${conversationId}/save`, {
         method: 'POST',
         headers: { 'Content-Type': 'application/json' },
         body: JSON.stringify({ messages }),
       });
     } catch (error) {
       console.error('Failed to save conversation:', error);
     }
   }, [agentName, conversationId, messages]);

   // 对话完成后调用
   useEffect(() => {
     if (!isRunning && messages.length > 0) {
       saveConversation();
     }
   }, [isRunning, saveConversation]);
   ```

2. **ConversationDrawer.tsx 修改** - 禁用缓存：
   ```typescript
   const loadConversations = async () => {
     setIsLoading(true);
     try {
       const res = await fetch(`/api/agents/${agentName}/conversations`, {
         cache: 'no-store',  // 禁用缓存
         headers: {
           'Cache-Control': 'no-cache',
         },
       });
       const data = await res.json();
       setConversations(data.conversations || []);
     } finally {
       setIsLoading(false);
     }
   };
   ```

3. **后端API增强** - 添加时间戳：
   ```python
   @app.get("/api/agents/{name}/conversations")
   async def list_conversations(name: str):
       conversations = conversation_manager.list_conversations(name)
       return {
           "conversations": conversations,
           "total": len(conversations),
           "last_updated": datetime.now().isoformat()  # 添加时间戳
       }
   ```

### 3.4 验收标准

| 编号 | 验收条件 |
|------|----------|
| AC-3.1 | 用户A在智能体test001中完成对话后，用户B刷新页面能在test001中看到该会话 |
| AC-3.2 | 对话完成后，后端 `data/conversations/{agent_name}/` 目录下有对应的JSON文件 |
| AC-3.3 | 会话列表API每次返回最新的数据（不被浏览器缓存） |
| AC-3.4 | 刷新页面后，当前会话的历史消息能正确恢复 |
