## 问题一：会话记录串台问题

### 1.1 根因分析

前端状态管理存在以下问题：

1. **page.tsx**: `currentConversationId` 和 `currentConversationMessages` 是全局状态，切换智能体时未重置
2. **ConversationDrawer.tsx**: `conversations` 状态在切换智能体后未及时清空，导致短暂显示旧数据
3. **AgentChat.tsx**: 内部状态（messages、fileContext等）在 agentName 变化时未重置

### 1.2 技术方案

#### 1.2.1 page.tsx 修改 (REQ-1.1)

**位置**: `frontend/src/app/page.tsx` 第474-495行

**修改内容**:
```typescript
const openAgentConfig = async (name: string) => {
  try {
    // REQ-1.1: 切换智能体时重置会话状态，避免会话串台
    setCurrentConversationId(null);
    setCurrentConversationMessages([]);

    const res = await fetch(`${API_BASE}/agents/${name}`);
    // ... 其余逻辑
  }
};
```

**原理**: 在用户点击切换智能体时，立即清空当前会话 ID 和消息列表，确保不会将上一个智能体的会话数据带入新的智能体。

#### 1.2.2 AgentChat.tsx 修改 (REQ-1.3)

**位置**: `frontend/src/components/AgentChat.tsx` 第170-183行

**修改内容**:
```typescript
// REQ-1.3: 监听 agentName 变化，重置内部状态
const prevAgentNameRef = useRef(agentName);
useEffect(() => {
  if (prevAgentNameRef.current !== agentName) {
    // agentName 发生变化，重置所有状态
    setMessages([]);
    setFileContext({ file_ids: [], file_infos: [] });
    setInputValue('');
    setHasError(false);
    setPendingFiles([]);
    prevAgentNameRef.current = agentName;
  }
}, [agentName]);
```

**原理**: 使用 `useRef` 跟踪上一个 agentName 值，当检测到变化时重置所有内部状态。这种方法比直接在依赖数组中添加 `agentName` 更可控，可以精确控制重置时机。

#### 1.2.3 ConversationDrawer.tsx 修改 (REQ-1.2)

**位置**: `frontend/src/components/ConversationDrawer.tsx` 第37-70行

**修改内容**:
```typescript
// REQ-1.2: 跟踪上一个 agentName，在变化时先清空会话列表
const prevAgentNameRef = useRef<string | null>(null);

useEffect(() => {
  if (open && agentName) {
    // REQ-1.2: agentName 变化时先清空，避免显示旧数据
    if (prevAgentNameRef.current !== agentName) {
      setConversations([]);
      prevAgentNameRef.current = agentName;
    }
    loadConversations();
  }
}, [open, agentName]);
```

**原理**: 在 agentName 变化时，先清空 `conversations` 状态，然后再发起加载请求。这样可以避免在加载新数据的过程中短暂显示上一个智能体的会话列表。

## 问题三：历史会话共享问题

### 3.1 根因分析

1. **浏览器缓存**: 前端 fetch 请求可能被浏览器缓存，导致获取的不是最新数据
2. **自动保存逻辑**: 已存在但需要确认正常工作

### 3.2 技术方案

#### 3.2.1 ConversationDrawer.tsx 修改 (REQ-3.2)

**位置**: `frontend/src/components/ConversationDrawer.tsx` 第52-70行

**修改内容**:
```typescript
// REQ-3.2: 加载会话列表时禁用浏览器缓存
const loadConversations = async () => {
  setIsLoading(true);
  try {
    const res = await fetch(`/api/agents/${agentName}/conversations`, {
      cache: 'no-store',  // 禁用缓存，确保每次都获取最新数据
      headers: {
        'Cache-Control': 'no-cache',
        'Pragma': 'no-cache',
      },
    });
    const data = await res.json();
    setConversations(data.conversations || []);
  } catch (error) {
    console.error("Failed to load conversations:", error);
  } finally {
    setIsLoading(false);
  }
};
```

**原理**:
- `cache: 'no-store'`: 告诉浏览器不要缓存响应
- `Cache-Control: no-cache`: 告诉服务器/代理不要使用缓存
- `Pragma: no-cache`: HTTP/1.0 兼容性头部

#### 3.2.2 自动保存逻辑确认 (REQ-3.1)

**位置**: `frontend/src/components/AgentChat.tsx` 第720-734行

## 测试建议

### 手动测试步骤

1. **会话隔离测试**:
   - 创建智能体 A 和 B
   - 在 A 中发送消息，创建会话
   - 切换到 B，确认聊天区域为空
   - 打开会话抽屉，确认显示为空（或仅显示 B 的历史会话）
   - 切换回 A，确认会话正常显示

2. **历史共享测试**:
   - 在浏览器 A 中完成对话
   - 在浏览器 B 中打开同一智能体
   - 刷新页面，确认能看到刚才的会话

### 注意事项

1. **useRef 跟踪模式**: 使用 `useRef` 跟踪上一个值是一种常见的 React 模式，可以避免在 useEffect 依赖数组中添加额外状态导致的循环更新问题。

2. **禁用缓存的影响**: `cache: 'no-store'` 会增加服务器负载，因为每次请求都会发送到服务器。但对于会话列表这种需要实时性的数据，这是必要的代价。

3. **状态重置顺序**: 在 page.tsx 中，会话状态的重置放在加载配置之前，确保即使加载失败也能清空旧状态。
