## 1. 背景与问题分析

### 1.1 用户反馈

End User反馈：之前迭代做了对话文件上传、Skill独立运行环境的功能，但实际使用后发现**基本不可用**：
- 文件看似可以上传
- 智能体也看似可以读取
- 但**实际调用Skill时失败**

### 1.2 现有架构问题诊断

经过代码审视，发现以下**核心问题**：

#### 问题1：AgentEngine 与 ExecutionEngine 未集成

```
AgentEngine (agent_engine.py)
    └── SkillTool.load_skill()  --> 仅返回技能文档文本（RAG方式）
    └── 无 execute_skill 工具调用能力

ExecutionEngine (execution_engine.py)
    └── execute_script() 方法存在
    └── 但从未被 AgentEngine 调用
    └── 仅通过 REST API `/api/agents/{name}/execute` 暴露
```

**问题本质**：LLM 没有途径通过 Tool Call 触发脚本执行，只能"看到"技能文档，无法真正运行脚本。

#### 问题2：SkillTool 中 execute_skill_tool 未注册到 LLM

```python
# skill_tool.py 第102-166行
def get_execute_tool_definition(self) -> Optional[Dict[str, Any]]:
    """返回脚本执行工具定义"""
    if not self.execution_engine:
        return None  # <-- execution_engine 从未被注入
```

`SkillTool` 初始化时需要 `execution_engine` 参数，但 `AgentEngine` 初始化 `SkillTool` 时**未传入**：

```python
# agent_engine.py 第52-59行
if skill_registry and config.skills:
    self.skill_tool = SkillTool(
        skill_registry=skill_registry,
        skills_dir=self.skills_dir,
        enabled_skills=config.skills
        # 缺失: execution_engine=...
        # 缺失: agent_name=...
    )
```

#### 问题3：文件上传与技能执行的数据流断裂

```
前端上传文件 --> FileStorageManager 存储
                    ↓
              backend.py 构建 file_context (仅文本描述)
                    ↓
              AgentEngine.stream() 将路径写入 system_prompt
                    ↓
              LLM 只看到文本，无法访问实际文件
                    ↓
              调用 execute_skill 时，input_file_ids 未被传递
```

#### 问题4：AgentManager 未持有 ExecutionEngine 实例

```python
# agent_manager.py - AgentInstance 初始化
self.engine = AgentEngine(
    self.config,
    self.mcp_manager,
    self.skill_registry,
    self.skills_dir,
    self.model_service_registry
    # 缺失: execution_engine
)
```

## 2. 需求目标

### 2.1 核心目标

实现一个**完整可用的** Agent 独立运行环境与文件处理系统：

1. **独立环境**：每个 Agent 拥有独立的 Conda 虚拟环境
2. **技能加载**：会话开始时自动加载用户勾选的 Skill
3. **脚本执行**：LLM 可通过 Tool Call 执行 Skill 脚本
4. **文件传递**：上传的文件能正确传递到脚本执行环境
5. **环境隔离**：依赖和数据互不影响

### 2.2 非目标

- RAG 方案（纯文档检索）- 彻底废弃
- 容器化方案（Docker）- 作为未来扩展，本次不实现

## 3. 功能需求

### 3.1 F1: Agent 独立运行环境

#### F1.1 环境生命周期管理

| 操作 | 触发时机 | 行为 |
|------|----------|------|
| 创建 | Agent 首次执行 Skill 脚本 | 调用 `conda create -p env_{agent_name} python=3.11` |
| 获取 | 会话开始 | 调用 `get_or_create_environment()` |
| 删除 | Agent 被删除 | 调用 `conda env remove -p env_{agent_name}` |

#### F1.2 依赖管理

- 每个环境独立安装 Skill 所需的 Python 包
- 包列表存储在 `data/environments/{agent_name}.json`
- 支持 `pip install` 增量安装

### 3.2 F2: Skill 按需执行

#### F2.1 工具注册

Agent 需要向 LLM 注册 `execute_skill` 工具：

```json
{
  "name": "execute_skill",
  "description": "Execute a skill's Python script in an isolated Conda environment.",
  "input_schema": {
    "type": "object",
    "properties": {
      "skill_name": {"type": "string"},
      "script_name": {"type": "string", "default": "main.py"},
      "arguments": {"type": "array", "items": {"type": "string"}},
      "input_file_ids": {"type": "array", "items": {"type": "string"}},
      "timeout": {"type": "integer", "default": 60}
    },
    "required": ["skill_name"]
  }
}
```

#### F2.2 执行流程

```
LLM Tool Call: execute_skill(skill_name="AB-pdf", input_file_ids=["abc123"])
    ↓
AgentEngine._execute_tool()
    ↓
SkillTool.execute_script()
    ↓
ExecutionEngine.execute_script()
    ↓
EnvironmentManager.get_or_create_environment()
    ↓
EnvironmentManager.execute_in_environment()
    ↓
返回执行结果给 LLM
```

### 3.3 F3: 文件上传与传递

#### F3.1 上传流程

```
用户选择文件 --> 前端 pendingFiles
    ↓
用户发送消息 --> POST /api/agents/{name}/files (multipart)
    ↓
FileStorageManager.upload_file() --> 返回 file_id
    ↓
POST /api/agents/{name}/chat/stream (file_ids: [...])
    ↓
执行时通过 input_file_ids 传递到脚本
```

#### F3.2 文件到脚本的数据传递

```python
# ExecutionEngine._prepare_work_dir_with_skill() 需要调用
await self._copy_input_files(agent_name, input_file_ids, work_dir)
# 文件被复制到 work_dir/input/ 目录
# 脚本通过相对路径 ./input/ 访问
```

### 3.4 F4: 前端文件上传组件

#### F4.1 UI 位置

- 调试对话窗口的输入框左侧
- 显示已选文件预览
- 支持点击上传和拖拽上传

#### F4.2 支持的文件类型

| 类型 | 扩展名 | MIME Type |
|------|--------|-----------|
| PDF | .pdf | application/pdf |
| Word | .docx, .doc | application/vnd.openxmlformats-officedocument.wordprocessingml.document |
| Excel | .xlsx, .xls | application/vnd.openxmlformats-officedocument.spreadsheetml.sheet |
| 文本 | .txt, .csv, .json | text/plain, text/csv, application/json |

#### F4.3 文件大小限制

- 单文件最大: 100MB
- 单次上传最大: 3个文件

## 4. 非功能需求

### 4.1 性能要求

| 指标 | 要求 |
|------|------|
| 环境创建 | 首次 < 60s，复用 < 100ms |
| 脚本执行超时 | 默认 60s，可配置 10-300s |
| 文件上传 | 100MB 文件 < 10s |

### 4.2 安全要求

- 脚本在独立 Conda 环境执行，不访问系统 Python
- 文件存储在 `data/files/{agent_name}/` 隔离目录
- Agent 删除时清理对应环境和文件

### 4.3 可用性要求

- 错误信息友好展示
- 脚本执行失败时返回 stderr
- 环境创建失败时提供重试机制

## 8. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Conda 环境创建慢 | 用户体验差 | 首次创建后复用，后台预热 |
| 脚本执行超时 | 任务失败 | 提供超时配置，异步执行长任务 |
| LLM 不调用工具 | 功能失效 | 优化工具描述，提供 few-shot 示例 |
| 文件过大 | 内存溢出 | 限制文件大小，流式处理 |
