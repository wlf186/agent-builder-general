## 一、需求背景与目标

### 1.1 问题背景

当前 Agent Builder 平台的文件上传与技能执行功能存在以下核心问题：

| 问题编号 | 问题描述 | 影响范围 | 严重程度 |
|---------|---------|---------|---------|
| P1 | Skill 没有可执行入口脚本（main.py） | AB-pdf、AB-docx 等 Skill 只有分散的工具脚本，LLM 无法正确调用 | 严重 |
| P2 | 文件上下文与执行脱节 | `file_context` 只是纯文本描述，`file_ids` 未注入到 `execute_skill` 工具 | 严重 |
| P3 | 依赖安装机制缺失 | 环境创建只安装基础 Python，没有 `requirements.txt` 自动安装 | 中等 |
| P4 | 工具描述不够明确 | `execute_skill` 工具描述过于技术化，LLM 难以理解如何正确调用 | 中等 |

### 1.2 目标

**核心目标**: 实现用户上传文件后，Agent 能够自动调用对应 Skill 的脚本处理文件，并返回处理结果。

**成功标准**:
1. 用户上传 PDF 文件后，Agent 能够自动调用 AB-pdf 技能处理
2. 用户上传 DOCX 文件后，Agent 能够自动调用 AB-docx 技能处理
3. 脚本执行在隔离的 Conda 环境中进行，依赖自动安装
4. 整个流程对用户透明，体验流畅

## 二、功能点拆解

### 2.1 环境管理模块

#### 2.1.1 功能描述

为每个 Agent 创建独立的 Conda 虚拟环境，支持自动安装 Skill 所需的依赖包。

#### 2.1.2 需求明细

| 需求ID | 需求描述 | 验收标准 | 优先级 |
|--------|---------|---------|--------|
| ENV-001 | 创建环境时检测 Skill 的 requirements.txt | 若 Skill 目录下有 `scripts/requirements.txt`，自动安装依赖 | P0 |
| ENV-002 | 环境创建后自动安装 Skill 依赖 | 首次执行 Skill 时，检查并安装缺失依赖 | P0 |
| ENV-003 | 支持多 Skill 依赖共存 | 不同 Skill 的依赖可以在同一环境中共存 | P1 |
| ENV-004 | 依赖安装失败时的优雅降级 | 安装失败时返回明确错误信息，不阻塞其他 Skill | P1 |

#### 2.1.3 数据模型

```python
class AgentEnvironment:
    agent_name: str
    environment_type: EnvironmentType  # CONDA
    status: EnvironmentStatus  # CREATING, READY, ERROR
    python_version: str
    packages: List[str]  # 已安装的包列表
    installed_dependencies: Dict[str, List[str]]  # {skill_name: [依赖列表]}
    created_at: str
    updated_at: str
    error_message: Optional[str]
```

### 2.2 Skill 执行模块

#### 2.2.1 功能描述

提供 Skill 脚本的标准化执行入口，支持参数传递和文件输入。

#### 2.2.2 需求明细

| 需求ID | 需求描述 | 验收标准 | 优先级 |
|--------|---------|---------|--------|
| SKILL-001 | 为 AB-pdf 添加 main.py 入口脚本 | 支持 `python main.py <input_file> [--output <output_file>]` | P0 |
| SKILL-002 | 为 AB-docx 添加 main.py 入口脚本 | 支持标准化的命令行参数 | P0 |
| SKILL-003 | main.py 支持 --help 参数 | 返回使用说明 | P1 |
| SKILL-004 | 脚本执行超时控制 | 默认 60 秒超时，可配置 | P1 |
| SKILL-005 | 执行结果标准化返回 | 返回 JSON 格式：`{status, output, error, files}` | P1 |

#### 2.2.3 main.py 规范

```python
#!/usr/bin/env python3
"""
AB-pdf Skill - PDF 文件处理

Usage:
    python main.py <input_file> [--action <action>] [--output <output_file>]

Actions:
    extract_text  - 提取 PDF 文本内容
    extract_forms - 提取表单字段
    fill_form     - 填充表单字段
    convert_images - 转换为图片

Example:
    python main.py ./input/document.pdf --action extract_text
"""

import argparse
import json
import sys
from pathlib import Path

def main():
    parser = argparse.ArgumentParser(description='AB-pdf: PDF processing skill')
    parser.add_argument('input_file', help='Input PDF file path')
    parser.add_argument('--action', default='extract_text',
                        choices=['extract_text', 'extract_forms', 'fill_form', 'convert_images'])
    parser.add_argument('--output', help='Output file path')
    parser.add_argument('--data', help='JSON data for form filling')
    args = parser.parse_args()

    # 处理逻辑...

    result = {
        "status": "success",
        "action": args.action,
        "output": "...",
        "files": []  # 生成的输出文件列表
    }
    print(json.dumps(result, ensure_ascii=False))

if __name__ == "__main__":
    main()
```

### 2.3 文件上传模块

#### 2.3.1 功能描述

支持用户上传文件，并将文件信息传递给 Agent 作为执行上下文。

#### 2.3.2 需求明细

| 需求ID | 需求描述 | 验收标准 | 优先级 |
|--------|---------|---------|--------|
| FILE-001 | 前端文件上传组件 | 支持 PDF/DOCX/XLSX 文件拖拽上传 | P0 |
| FILE-002 | 文件上传后返回 file_id | 前端获取 file_id 用于后续调用 | P0 |
| FILE-003 | file_context 包含 file_ids | Agent 感知上传的文件 ID 列表 | P0 |
| FILE-004 | execute_skill 自动注入 file_ids | LLM 调用 execute_skill 时自动携带 file_ids | P0 |
| FILE-005 | 文件预览功能 | 上传后显示文件名、大小、类型 | P1 |

#### 2.3.3 文件上传流程

```
用户选择文件 → 前端上传 → 后端存储 → 返回 file_id
      ↓
前端更新 file_context（包含 file_ids）
      ↓
用户发送消息 → Agent 感知 file_context
      ↓
Agent 调用 execute_skill（自动携带 file_ids）
      ↓
ExecutionEngine 复制文件到工作目录 ./input/
      ↓
脚本执行，处理 ./input/ 下的文件
```

|--------|---------|---------|--------|
| TOOL-001 | 添加使用场景说明 | 描述何时应该调用此工具 | P0 |
| TOOL-002 | 添加调用示例 | 提供具体的 JSON 调用示例 | P0 |
| TOOL-003 | 参数说明更友好 | 用自然语言描述每个参数的作用 | P1 |
| TOOL-004 | 自动提示可用 Skill | 根据上传文件类型推荐 Skill | P1 |

## 三、验收标准

### 3.1 功能验收

| 场景 | 操作步骤 | 预期结果 |
|------|---------|---------|
| PDF 文本提取 | 1. 上传 PDF 文件 2. 发送"提取这个 PDF 的文本" | Agent 调用 execute_skill，返回 PDF 文本内容 |
| PDF 表单填充 | 1. 上传 PDF 表单 2. 发送"帮我填充表单，姓名填张三" | Agent 调用 execute_skill，返回填充后的 PDF |
| DOCX 文档创建 | 1. 发送"创建一个包含标题'测试'的 Word 文档" | Agent 调用 execute_skill，返回创建的 DOCX 文件 |
| 依赖自动安装 | 1. 首次使用 AB-pdf 2. 环境自动安装 PyPDF2 等依赖 | 执行成功，无依赖缺失错误 |

### 3.2 性能验收

| 指标 | 要求 |
|------|------|
| 文件上传响应时间 | < 2 秒（10MB 以内文件） |
| 环境创建时间 | < 60 秒（含依赖安装） |
| 脚本执行超时 | 默认 60 秒，可配置 |
| 并发执行数 | 每个 Agent 最多 3 个并发任务 |

### 3.3 兼容性验收

| 环境 | 要求 |
|------|------|
| 浏览器 | Chrome 90+, Firefox 88+, Safari 14+ |
| Python | 3.11+ |
| Conda | Miniconda3 / Anaconda3 |

## 五、风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Conda 环境创建失败 | Skill 无法执行 | 提供降级方案：使用系统 Python |
| 依赖安装冲突 | 部分 Skill 不可用 | 隔离不同 Skill 的依赖版本 |
| 大文件上传超时 | 用户体验差 | 增加上传进度提示，支持断点续传 |
| 脚本执行卡死 | 资源占用 | 强制超时终止，资源清理 |
