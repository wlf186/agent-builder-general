# 技术方案选型报告 - Skill执行状态显示异常

## 问题背景

客户反馈使用 `skill-test-pdf` 和 `skill-test-doc` 智能体时，"技能执行状态"同时显示"执行完成"和"执行失败"。

## 问题定位过程

### 1. 数据配置检查
- **agent_configs.json**: 智能体配置 `skills: ["ab-pdf"]` / `skills: ["ab-docx"]`（小写）
- **skills_index.json**: 注册的技能名称为 `"ab-pdf"` / `"ab-docx"`（小写）
- **结论**: 配置数据无重复，非历史残留问题

### 2. 后端事件生成逻辑检查 (`src/agent_engine.py`)

**关键代码位置**: 第1189-1218行

```python
# 检查是否是 skill 工具
is_skill_tool = tool_name == SkillTool.TOOL_NAME or tool_name == SkillTool.EXECUTE_TOOL_NAME

# 如果是 skill 工具，发送 skill_loading 事件
if is_skill_tool:
    yield {"type": "skill_loading", "skill_name": skill_name}

# 执行工具
result = await self._execute_tool(tool_name, tool_args)

# 如果是 skill 工具，发送 skill_loaded 事件
if is_skill_tool:
    yield {"type": "skill_loaded", "skill_name": skill_name, "success": not result.startswith("Error:")}

# 输出工具结果
yield {"type": "tool_result", "name": tool_name, ...}
```

**发现**: `is_skill_tool` 包含两个工具：
- `SkillTool.TOOL_NAME = "load_skill"` - 加载技能说明文档
- `SkillTool.EXECUTE_TOOL_NAME = "execute_skill"` - 执行技能脚本

### 3. 前端状态管理逻辑检查 (`frontend/src/components/AgentChat.tsx`)

**skill_loading 事件处理** (第616-640行):
```typescript
streamingSkillStatesRef.current = [...streamingSkillStatesRef.current, newSkillState];
// ❌ 每次都追加新状态，不检查是否已存在同名skill
```

**skill_loaded 事件处理** (第641-675行):
```typescript
streamingSkillStatesRef.current = streamingSkillStatesRef.current.map(s =>
  s.skillName === skillName ? { ...s, status: success ? 'completed' : 'failed' } : s
);
// ⚠️ 只更新第一个匹配的同名状态
```

**tool_result 事件处理** (第571-577行):
```typescript
skillStates: toolName === 'execute_skill'
  ? msg.skillStates?.map(s =>
      s.status === 'executing'
        ? { ...s, status: isError ? 'failed' : 'completed' }
        : s
    )
  : msg.skillStates
// ⚠️ 只更新 status='executing' 的状态
```

## 根因分析结论

### 核心问题
**前端 `skillStates` 数组允许存在同名skill的多个状态项，但更新逻辑不完整。**

### 问题场景复现

当LLM处理用户请求时，可能按以下顺序调用工具：

1. **调用 `load_skill`**（加载技能说明）
   - → `skill_loading` → 添加状态项: `{skillName: "ab-pdf", status: "loading"}`
   - → `skill_loaded` → 更新为: `{skillName: "ab-pdf", status: "completed"}`

2. **调用 `execute_skill`**（执行脚本）
   - → `skill_loading` → **又添加**状态项: `{skillName: "ab-pdf", status: "loading"}`
   - → `skill_loaded` → 更新第二个状态项: `{skillName: "ab-pdf", status: "completed"}`

**结果**: `skillStates` 数组中有两个同名skill的状态项！

### 状态显示冲突来源

如果：
- `load_skill` 返回结果包含 "Error" → `skill_loaded` 判断失败 → status = "failed"
- `execute_skill` 执行成功 → `skill_loaded` 判断成功 → status = "completed"

前端UI会显示：
- ✓ ab-pdf 执行完成
- ✗ ab-pdf 执行失败

## 修复方案建议

### 方案A: 前端去重修复（推荐，影响最小）

**修改文件**: `frontend/src/components/AgentChat.tsx`

**修改思路**: 在追加新状态前，检查是否已存在同名skill，存在则更新而非追加

```typescript
// skill_loading 事件处理
const existingIndex = streamingSkillStatesRef.current.findIndex(s => s.skillName === skillName);
if (existingIndex >= 0) {
  // 已存在，更新状态
  streamingSkillStatesRef.current = streamingSkillStatesRef.current.map((s, i) =>
    i === existingIndex ? { ...s, status: 'loading', message: '正在加载...' } : s
  );
} else {
  // 不存在，追加新状态
  streamingSkillStatesRef.current = [...streamingSkillStatesRef.current, newSkillState];
}
```

**优点**:
- 修改范围小，只涉及前端
- 不影响后端事件生成逻辑
- 向后兼容

**缺点**:
- 如果同一个skill需要被多次并行调用（罕见场景），状态会混淆

### 方案B: 前端使用call_id去重（更严谨）

**修改文件**: `frontend/src/components/AgentChat.tsx`

**修改思路**: 使用 `call_id` 作为唯一标识，而非 `skillName`

```typescript
// 状态结构改为
interface SkillExecutionState {
  callId: string;  // 新增
  toolName: string;  // "load_skill" 或 "execute_skill"
  skillName: string;
  status: 'loading' | 'executing' | 'completed' | 'failed';
  message: string;
}
```

**优点**:
- 精确追踪每次工具调用
- 同一skill可被多次调用而不混淆
- 可区分 load_skill 和 execute_skill 的状态

**缺点**:
- 需要修改状态结构和UI渲染逻辑
- 改动范围较大

### 方案C: 后端事件分离（架构级优化）

**修改文件**: `src/agent_engine.py`

**修改思路**: 为 `execute_skill` 使用独立的事件类型

```python
if tool_name == SkillTool.TOOL_NAME:
    yield {"type": "skill_loading", "skill_name": skill_name}
elif tool_name == SkillTool.EXECUTE_TOOL_NAME:
    yield {"type": "skill_executing", "skill_name": skill_name}  # 新事件类型
```

**优点**:
- 语义更清晰
- 前端可分别处理两种操作

**缺点**:
- 需要同时修改前后端
- 增加事件类型复杂度

## 推荐方案

**采用方案A**（前端去重修复），原因：
1. 修改范围最小，风险最低
2. 实际使用中，同一skill很少需要并行多次调用
3. 可快速修复当前问题
